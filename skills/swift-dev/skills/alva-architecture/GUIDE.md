# ALVA — Agent-Legible Vertical Architecture

The cost function: minimize **tokens-until-a-correct-verifiable-change**. Every rule below is a
consequence of that one ruler. Full portable doctrine (publishable standalone, no akios
dependency): `alva-architecture-doctrine.md`. This guide is the akios-specific realization —
read it, don't re-derive the doctrine from scratch each time.

## The slice tree

A feature is a **vertical slice** — everything it needs lives in one directory, in the exact
same shape every other feature uses:

```
Project/
  Router/                    → navigation + custom-flow coordinators
  Container/                 → DI: composition root + factories
  Foundation/
    Design-tokens/           → DesignSystem struct, role ViewModifiers, native-first components
    Code-tokens/             → shared protocols + helpers (behind a contract)
    usage-ledger.json        → deterministic promotion/demotion counts — read-only to you
  Features/
    <Feature>/
      domain/                → entities + use cases (the slice owns them)
      data/                  → repository protocol + concrete(s)
      presentation/          → per-view folders + Models/
        <View>/               { <View>.swift, <View>Model.swift, components/<Component>/ }
        Models/               { JIT view-state DTOs, slice-local }
      contract/               → public facade: protocol + DTOs other slices import
      tests/                  → co-located TDD
      Feature-spec.md         → intention + declared contract/Foundation consumption
  scratchs/                  → rejected ui-variations rounds, compilable + previewable,
                                out of the app target
```

**Uniformity is the point.** Every feature has *exactly* this shape — no per-feature judgment
calls about where things go. That repetition is what lets you load one slice and correctly
predict the shape of the next one without re-deriving it.

## The one hard rule: contract boundary

A feature imports another feature's `contract/` and `Foundation/` — **never** its `domain/` or
`data/` internals. If you find yourself importing another slice's internals, the contract is
wrong (add what you need to it) or the screen is in the wrong slice (see "Screen ownership"
below). This boundary is enforced by lint in `code-review` and the pre-commit hook, graduating
to compiler-enforced local SPM modules only when the app has outgrown lint (a recurring
violation, or the user asks) — not a day-one tax on a 3-screen app.

**Screen ownership.** A screen spanning two domains belongs to the feature that owns the
*intention*; it consumes the other domain via `contract/`. "My purchases" → intention is
Purchase → lives in `Purchase/`, consumes `User.contract`. "Profile with recent purchases" →
intention is Profile → lives in `User/`, consumes `Purchase.contract`.

## Foundation: consult before you create

**Before writing any new helper, protocol, or shared component, consult only `Foundation/`** —
never search the whole repo for "is this already done somewhere." `Foundation/` is small and
indexed; that's the entire point.

- **Never count usage yourself.** `Foundation/usage-ledger.json` is generated by a
  deterministic tool (git-hook + ripgrep, or a compiler-index script). You **read** it; you
  never estimate "this is probably used elsewhere" by grepping.
- **A `candidates_promote` entry becomes a task**, not a silent move. Promotion is *suggested*,
  reviewed like any other task — high blast-radius, near-irreversible changes don't happen
  inside a run unnoticed.
- **Design-tokens vs Code-tokens have different bars.** Design-tokens (components, view
  modifiers, UI enums — pure, leaf, ~zero blast radius) promote liberally, threshold as low as
  2 uses. Code-tokens (protocols, helpers, services — carry behavior and dependencies) promote
  rarely and go up **behind a contract**, not just moved to a new folder — promoting a service
  declares it core domain.

## Carry-over laws (still apply, now inside `presentation/<View>/`)

These didn't change — they just live one level deeper than a shared `PresentationLayer/` would
have put them:

- **Build order:** components → dumb screen → make-it-live. Build the leaf pieces first.
- **Dumb components:** a component receives data + closures via `init`; it has no knowledge of
  where its data comes from or what its actions do. It's born inside the view that first needed
  it (`presentation/<View>/components/<Component>/`) and promotes to `Foundation/Design-tokens/`
  via the ledger on its 2nd distinct use — no intermediate feature-wide tier.
  - **VM-per-screen; few screens, many components** — a screen's ViewModel is its own; reusable
    UI comes from breaking out components, not from sharing ViewModels.
- **JIT view-state DTOs** live in `presentation/Models/`, slice-local until a second slice
  needs one (then it's a Foundation or contract question, not an automatic promotion).
- **DesignSystem token struct, native-over-custom budget, Nielsen heuristics, text/image role
  modifiers, `containerRelativeFrame` adaptivity** — unchanged doctrine, homed in
  `Foundation/Design-tokens/` and enforced at the `align-ui` design phase + review.
- **Explore + remix (`ui-variations`)** is orthogonal to structure: an approved variation
  graduates directly into `presentation/<View>/`; rejected rounds archive to top-level
  `scratchs/<Component-or-View>.swift`, compilable and previewable but outside the app target.

## Composition: Router + Container

Only the top of the project knows more than one feature.

- **`Container/`** is the composition root: it registers factories and injects contracts
  between features (e.g. a `SquadListViewModelFactory`).
- **`Router/`** resolves navigation. A simple push/present is a `navigationDestination` the
  Router resolves. A **multi-step custom flow** (onboarding, a purchase wizard spanning several
  screens and slices) gets a named **coordinator** object in `Router/` that owns the flow's
  state machine and draws slice screens via the Container's factories — it's a
  composition-root citizen, so it's the one place allowed to know several slices. It still
  only *consumes* those slices through their `contract/`. Reach for a coordinator only for
  genuinely multi-step custom flows; plain Router navigation is the default.
- **DI/coordination never bypasses the design-token rule.** A view or component a coordinator
  draws through the Container's factories still consumes `Foundation/Design-tokens/` statically
  (`.textStyle`/`.imageStyle`, `DesignSystem.*`) — never through `@Environment` or a service the
  coordinator injects. See `skills/swift-dev/skills/swiftui-design-system/GUIDE.md` for the
  token/role-modifier convention; multi-step flow orchestration and visual-leaf consumption are
  orthogonal concerns.

## Worked example — Squad slice (futebol-manager)

`Features/Squad/` owns `domain/{Player,Squad,Match}`, `data/{PlayerRepository protocol +
SwiftData concrete}`, `presentation/{SquadList/{SquadListView.swift, SquadListViewModel.swift,
components/PlayerRow/, components/FormBadge/}, Models/SquadRowData}`, `contract/{SquadContract
protocol + DTOs}`, `tests/`, and a `Feature-spec.md` declaring it consumes
`Foundation.DesignSystem` + `Match.contract`. `Container/` registers a
`SquadListViewModelFactory`; `Router/` resolves `.squadList` via that factory. A "sign a new
player" wizard spanning Squad + Wallet gets a `SignPlayerCoordinator` in `Router/`. When
`FormatCurrency` shows up in Squad, Wallet, and Invoice, the ledger flags it and a promotion
task moves it to `Foundation/Code-tokens` behind a contract.

## When this doesn't apply

A single-slice app degrades gracefully: one `Features/<F>/`, an almost-empty `Foundation/`, no
boundary to enforce yet. The structure imposes ~zero tax until a second slice appears — don't
force `Router/`/`Container/` ceremony onto a genuinely single-screen app; scale the structure to
the number of features that actually exist.
