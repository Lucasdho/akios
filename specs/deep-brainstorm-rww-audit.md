# akios — deep-brainstorm R-W-W Validation Audit (Phase 5)
**Working spec · v1.0**

> This spec describes the addition of a post-burst R-W-W validation audit phase to the
> `deep-brainstorm` skill. After all specs are written (Phase 4 Spec-burst), Phase 5 scores
> each spec against Real / Win / Worth-It criteria, writes a consolidated `specs/rww-audit.md`
> in the target project, and sets Roadmap status for any weak specs to `needs-revision`.
> `just-vibes` is updated to skip `needs-revision` specs by default.
> Everything here is settled unless marked *open*.

> **State:** designed

---

## 1. What this adds

After Phase 4 (Spec-burst) writes all `specs/*.md` files, a new **Phase 5 — Validate** runs:

1. Scores each spec against 3 R-W-W dimensions (see §3).
2. Writes a consolidated **`specs/rww-audit.md`** file in the target project — one row per spec.
3. Updates `Roadmap.md` for any spec scoring below threshold:
   - **Red (0–40):** status → `needs-revision`; spec is not execution-ready.
   - **Yellow (41–70):** status stays `designed`; row gets `[audit: shaky]` note.
   - **Green (71–100):** status stays `designed`; no annotation.
4. The existing Phase 5 (Review & close) shifts to **Phase 6**.

**What this does NOT change:**
- Spec files themselves are not modified by the audit. Audit lives in `rww-audit.md` only.
- Phase 4 (Spec-burst) runs identically — audit is post-hoc, not during writing.
- The rubric is a signal, not a veto: a red spec can still be executed with `--force`.

---

## 2. Why separate audit file, not per-spec annotation

**Decision:** write `specs/rww-audit.md` as a consolidated audit dashboard rather than
appending a `## R-W-W Audit` section to each spec file.

**Reason:** A spec is a design document; audit metadata is operational governance. Mixing
them conflates two audiences (the spec author vs. the pipeline gate). The audit file is a
lightweight, scannable table. Keeping it separate also means re-running the audit (after
revising a weak spec) replaces one file, not N spec files.

**Rejected:** per-spec append (creates spec/audit drift when specs are revised) and both
(duplication that diverges over time).

---

## 3. Scoring rubric (adapted from FounderLens MVA for skill/feature specs)

The FounderLens R-W-W rubric was designed for product/market fit. For `deep-brainstorm`,
which produces specs for app features and infrastructure domains, the criteria are reframed:

| Dimension | Criterion | Max |
|---|---|---|
| **Real** | Does this domain address a genuine gap / pain in the app's user workflow? Is the scope grounded in Context.md / Vision.md / the Discover phase? | 30 |
| **Win** | Can this domain be built within the app's stated constraints (stack, team, no-external-deps rules)? Does it differentiate vs. the alternatives identified in Discover? | 40 |
| **Worth It** | Is the execution effort (estimated spec complexity) justified by the value to the user? Can it be decomposed into one `spec-to-tasks` session (not a multi-month project)? | 30 |

**Total:** 100.

**Bands:** 71–100 = Green (proceed) · 41–70 = Yellow (shaky, proceed with caution) · 0–40 = Red (needs revision before execution).

**Scoring posture:** honesty over flattery. A spec that covers infrastructure with no user-facing value scores low on Real. A spec that's unboundedly large scores low on Worth It. Weak specs are told they're weak — with a one-line remediation hint.

**Note:** In unattended (just-vibes) mode, scores are derived from the Discover phase
decisions + spec contents. In interactive mode, the user may adjust scores with a reason
(recorded in the audit file).

## 4. `specs/rww-audit.md` format

```markdown
# R-W-W Spec Audit
Generated by `/akios:deep-brainstorm` Phase 5.
Re-run manually: `/akios:deep-brainstorm audit` (open risk — see §7).

| Spec | Real /30 | Win /40 | Worth It /30 | Total | Band | Roadmap action | Note |
|---|---|---|---|---|---|---|---|
| onboarding.md | 28 | 35 | 25 | 88 | Green | — | |
| infra-auth.md | 22 | 30 | 20 | 72 | Green | — | |
| payments.md | 15 | 25 | 10 | 50 | Yellow | [audit: shaky] | Scope unbounded; split or descope |
| data-sync.md | 10 | 20 | 8 | 38 | Red | needs-revision | No grounded pain in Discover; revisit after user research |
```

The file is **written fresh on every deep-brainstorm run** (not appended). If a prior audit
exists, it is overwritten.

### New Phase 5 — Validate (R-W-W audit)

Insert after the current Phase 4 (Spec-burst) section, before Phase 5 (Review & close, which
becomes Phase 6):

```
## Phase 5 — Validate (R-W-W audit)

For each spec written in Phase 4, score against the R-W-W rubric (see §3 of the audit spec).
In interactive mode: show the table, explain any Red or Yellow scores, wait for acknowledgement.
In unattended mode: derive scores from Discover decisions + spec content; record reasoning.

Write `specs/rww-audit.md` (format: §4 of the audit spec).

Update `Roadmap.md`:
- Red specs: change status from `designed` to `needs-revision`.
- Yellow specs: append `[audit: shaky]` to the Notes column.
- Green specs: no change.

Announce the audit summary (X green, Y shaky, Z need revision) before moving to Phase 6.
```

### Rename Phase 5 → Phase 6

Change the heading `## Phase 5 — Review & close` to `## Phase 6 — Review & close`.
Update the phase number in the summary section within Phase 6 (references to "Phase 5"
become "Phase 6").

## 6. Changes to `skills/just-vibes/SKILL.md`

### Fuel detection: skip `needs-revision` by default

In the **Fuel detection procedure** (the numbered list), add a rule to step 3:

> When reading `Roadmap.md` for specs at status `designed`, **also skip any spec at status
> `needs-revision`** unless `--force` was passed. Log skipped specs in the journal with reason
> "audit: needs-revision".

### Fuel precedence table

The existing fuel list covers `designed` specs. Add a note:

> **`needs-revision` specs:** below-threshold R-W-W score; skipped in default mode.
> Pass `--force` to include them (at your own risk; audit findings apply).

### PARK logic

In step 4 (GATE), add: a spec in `needs-revision` that passes quality gate is still not
delivered; it must be revised (audit score updated) before merging. This prevents
accidentally shipping a spec the audit flagged as weak.

## 7. Open / next

- **[OPEN RISK — rubric subjectivity]** R-W-W scores are derived by the AI from spec content
  in unattended mode. There is no external ground truth. Scores reflect the Discover-phase
  decisions locked in the same session — if Discover was shallow, the audit will reflect that.
  Mitigation: audit file always shows the reasoning, not just the score. Human reviews after.
- **[OPEN RISK — re-audit command]** The audit doc says "re-run: `/akios:deep-brainstorm audit`"
  but that subcommand doesn't exist yet. Deferred; the user can re-run the full deep-brainstorm
  to regenerate, or edit `rww-audit.md` manually.
- **[EXPLICIT RISK — status `needs-revision` is new]** Adding a status between `designed` and
  `planned` means `workflow.yml` and any skill that pattern-matches on `designed` must be
  checked. Verified: just-vibes picks `designed` specs; the fix in §6 covers it. Any other
  skill that reads Roadmap.md status should be grepped.

## Worked example

User runs `/akios:deep-brainstorm` on a fitness app. Phase 4 writes 8 specs. Phase 5 scores:
5 Green, 2 Yellow (settings and analytics — real but low WTP signal), 1 Red (social-feed —
no grounded evidence of demand in the Discover phase). `rww-audit.md` is written. Roadmap
updates: `social-feed.md` → `needs-revision`; settings/analytics get `[audit: shaky]` notes.
Phase 6 shows the summary. User later revises the social-feed spec after doing user research,
manually changes status back to `designed`, re-runs audit (manually via editing), and the next
`/akios:just-vibes` picks it up.
